------ Intro ------


postman یک ابزار برای بررسی و تست api ها است. ما با این platform سایت یا Api نمی نویسیم و تنها Api هایی که داریم را تست می کنیم.

از آنجایی که postman برای تست Api ها است اصلا اهمیتی ندارد که با چه زبانی کد نویسی می کنید، به راحتی می توانید از این ابزار برای تست هر Api استفاده کنید.


postman امکان کار تیمی روی یک پروژه را به ما می دهد. ممکن است ما خودمان Api نویس باشیم یا این که برای مثلا برای سایت یا App اندورید خود از Api استفاده می کنیم. این اصلا اهمیتی ندارد و ما به راحتی در هر شرایطی می توانیم از این ابزار استفاده کنیم.


API Client ==> سیستمی که درخواست را به Api ارسال می کند. این می توان یک App اندرویید باشد یا یک سایت با فریم ورک های جاوااسکریپتی یا کلا ارسال کننده درخواست خود postman باشد.


postman زا restful - Soap - GraphQL پشتیبانی می کند.


پستمن همچنین امکان ایجاد workspace ها محتلف برای کارهای تیمی مختلف به صورت real-time را می دهد.


یک از ویژگی های postman این است که می توانیم Mock & Design طراحی کنیم یعنی یکسری end point تعریف کنیم و برنامه خود را براساس آن بسازیم و بعد این شکل طراحی را بدهیم به برنامه نویس backend و بگوییم که پاسخ های که از سرور برگشت داده می شود باید در این قالب باشند.

همچنین پستمن به ما امکان ایجاد Documention برای end point ها می دهد و ما به راحتی این doc را در اختیار برنامه نویس وب یا موبایل قرار می دهیم تا به راحتی بداند که end point ها چی هست و براساس آن ها برنامه نهایی و پاسخ های یا درخواست ها را طراحی کند.

------ Start ------


برای کار تیمی روی یک پروژه باید workspace ایجاد کنید.

 

برای پروژه های شخصی که تنها خودمان کار می کنیم می توانیم در هنگام ساخت نوع personal را انتخاب کنیم.

 

می توانیم برای هر پروژه یک workspace داشته باشیم یا اینکه یک workspace داشته باشیم و برای هر پروژه آن را پوشه بندی کنیم.

 

می توانیم خودمان workspace یا اینکه از نمونه دیفالتی که خودش قرار داده است استفاده کنیم. زمانی که روی اسم workspace مورد نظر کلیک کنیم وارد آن می شویم.

 

در هر workspace یک ایکون سبز sync وجود دارد که می گوید اطلاعات شما در حال ذخیره شدن روی سرور است و می توانید در هر جایی به آن ها دسترسی داشته باشید.

 

در بخش overview یکسری اطلاعات توضیحی به ما می دهد که زیاد کاریش نداریم. حال روی علامت + می زنیم و یک tab جدید ایجاد می کنیم.

 

این تب بخش اصلی ما برای تست API ها است. در این بخش یک URI و METHOD ست می کنیم و درخواست ها به آن ارسال می کند. یعنی خود postman به URL که دادیم با متود مورد نظر یک درخواست ارسال می کند و جواب را برمیگرداند و نمایش می دهد.

 

در پایین صفحه اگر از طریق اینترنت کار می کنیم باید مقدار را روی agent browser بگذاریم اما اگر روی desktop کار می کنیم باید این مقدار را desktop بگذاریم.


------ Collection ------


برای منظم کردن درخواست هایی که برای یک پروژه ارسال می کنیم، آن ها را داخل collection قرار می دهیم.

روی تب کناری روی collection می زنیم و بعد crate collection را انتخاب می کنیم. بعد اسم آن را وارد می کنیم. حالا کالکشن ما ساخته شده و بعد زمانیکه که URL را وارد کردیم و می توانیم روی Save بزنیم و اسم برای چیزی که ذخیره می شود امتخاب کنیم و کالکشن را هم  انتخاب کنیم.

با این کار تمام درخواست ها و API ها ما دسته بندی میشوند و دیگر نیازی به سردرگمی نیست.


------ Environment ------

 

گلوبال یعنی یک دسترسی یه یک چیز در کل کالکشن ها اما Environment یک محیط کوچک تر است که خودمان تعیین می کنیم چه چیزهایی داخل آن باشد.

برای اینجاد Environment روی چشم بالا می زنیم و گزینه add در بخش Environment را انتخاب می کنیم.


------ Variable ------


ممکن است تعداد زیادی درخواست داشته باشیم که به یک URL خاص ارسال می شوند و دامنه ارسالی در همه آنها یکسان است. حالا اگر فردا روزی دامنه سایت تغییر کرد ما مجبور می شویم که تمام URL ها را تغییر دهیم که اصلا کار معقولانه ای نیست. به همین دلیل postman  یک قابلیت با نام variable در اختیار ما قرار می دهید که به وسیله آن می توانیم موارد تکراری را تنها یکبار در قالب یک name تعریف کنیم و دفعات زیادی از آن استفاده کنیم. حالا اگر دامنه سایت تغییر کرد ما تنها باید آن variable را اصلاح کنیم.

 

برای ایجاد variable آن قسمتی از URL که در درخواست های مختلف تکراری است را select میکنیم. حالا خودش یک  pop-up باز می کند که داخل آن می گوید آیا می می خواهید set variable انجام دهید یا نه. اگر انتخاب کردیم لیست variable های قبلی را می آورد یا اینکه می توانیم یکی اضافه کنیم.

ابتدا یک نام انتخاب می کنیم و بعد نوع آن که یکی از مقادیر زیر است را انتخاب می کنیم.

Global ===> در تمام کالکشن ها و Environment ها به آن دسترسی داریم.

Environment  ==>  فقط در Environment انتخابی در دسترس است.

Collection ==> فقط در کالکشن انتخابی در دسترس است.

 

حالا هر کدام را زدیم در آن ذخیره می شود. اگر کالکشن را انتخاب کردیم برای دیدن آن باید روی کاکشن مورد نظر بزنیم و به بخش variables برویم. در این بخش دو مورد وجود دارند.

INITIAL VALUE     ==> یعنی اگر پروژه را برای کسی فرستادیم یا share این مورد نمایش داده می شود.

CURRENT VALUE    ==> یعنی مقدار اصلی که خودمان می توانیم ببینیم و کسی نباید به آن را ببیند زیرا ممکن است حاوی اطلاعات امنیتی باشد.

 

برای Environment   و Global هم به همین صورت است ولی برای مشاهده آن ها باید روی چشم بزنیم و Environment  یا Global مورد نظر برویم.

------ Path Variable ------


برای مثال زمانی که می خواهیم پست شماره 10 از سایر پست ها را درخواست کنیم URL ما به شکل posts/10 می شود. به 10 که متغییر است و محتوای مجزایی را فراخوانی می کند path variable می گویند.

برای اعمال این کار در post من ما در URL  ابتدا : قرار می دهیم و بعد اسم دلخواهی که دوست داریم. با این کار یک جدول خودش اضافه می کند با نام path variable که نام ما در آن قرار دارد هر مقداری برای آن ست کنیم به این منزله است که ما مثلا ایدی دادیم

/posts/12   ==   /posts/:post_id


------ Post Request ------


برای فرستادن این درخواست با post man ابتدا باید متود را روی post قرار دهیم و بعد در تب body مقادیر مورد نظر که باید ارسال شود را ست کنیم.

 

در قسمت body  ما چنتا موضوع داریم

 

none   ==>  یعنی چیزی قرار نیست همراه درخواست ارسال کنیم ومقداری قرار نیست به سمت سرور ارسال شود. در این صورت در بخش headers گزینه مربوط به content-type حذف میشود.

 

form-data  ===>  یعنی می توانیم در این درخواست فایل هم ارسال کنیم. تنها کافی است بخش text را به file تبدیل کنیم. هدر content-type در این مورد به multipart-formdata تبدیل می شود.

 

x-www-form-underscore  ===> در این مورد تنها می توانیم درخواست هایی از نوع text ازسال کنیم و هدر content-type برابر با x-www-form-underscore است.

 

raw   ==> برای زمانی است که می خواهیم به صورت دستی درخواست با فرمت Json یا فرمت دیگر که سرور پشتیبانی می کند ارسال کنیم. زمانی که این مورد را انتخاب می کنیم یک بخش اضافه می شود که در آن می توانیم فرمت درخواست ارسالی را تعیین کنیم که همزمان با آن هدر هم تغییر می کند. برای مثال json یا text 

 

binary ==> برای ارسال فایل ها به صورت باینری است

 

graphQL ===> برای درخواست هایی است که در سمت سرور به شکل QraphQl به آن ها رسیدگی می شود. ما در حال حاضر از Restful استفاده می کنیم.

------ PUT & Delete Method ------


برای update در پستمن می توانیم از متودهای PUT و PATCH استفاده کنیم. این دو تفاوتی ندارند و هر دو کار آپدیت را انجام می دهند. برای اپدیت علاوه بر اینکه باید دیتاهایی که باید به عنوان آپدیت ذخیره می شوند را ارسال کنیم، باید URL به شکل زیر باشد

PUT or PATCH    ||   {{baseURL}}/posts/:id

 

برای DELETE هم دقیقا مثل آپدیت عمل میکنیم با این تفاوت که دیگر مقداری نباید ارسال کنیم و تنها URL باید به شکل زیر باشد.

DELETE  ||    {{baseURL}}/posts/:id


------ Autoraization ------


گاهی در زمان استفاده از postman نیاز است درخواست های خود را به مسیرهایی ارسال کنیم که نیاز به login و ... یا نیاز به Autorization با توکن دارند. در این صورت ما باید همراه درخواست خود Token را هم ارسال کنیم تا سرور اجازه دسترسی به ما بدهد. برای این کار به روش زیر عمل می کنیم.

برای اینکار ابتدا توکن را که ساخته شده است ابتدا با درخواست لاگین برگردانیم و بعد آن را کپی کنیم. سپس در پستمن به بخش Autorization برویم و در select box گزینه Bearer token را انتخاب کنیم و بعد در اینپوت توکن خود را وارد کنیم. در واقع این کار برای ما یک header ست می کند که حاوی توکن است و به سرور می فهماند که من توکن را دارم.

بهتر است به جای وارد کردن توکن به صورت دستی در همه درخواست هایی که نیاز به Autorization دارند، این توکن را در قالب یک متغییر تعریف کنیم و به آن ها پاس دهیم تا نیاز نباشد هربار که توکن تغییر کرد همه آن ها را تغییر دهیم و تنها همان متغییر را تغییر دهیم.


------ Pre-request Script ------


در این بخش دستوراتی که باید قبل از ارسال request اجرا شوند را می نویسیم. در این بخش باید کدهای جاوااسکریپت بنویسیم.

هدف از این بخش این است که قبل از ارسال درخواست یکسری متغییرها را تغییر بدهد.

خود پستمن برای این کار یک سری تابع در کنار صفحه قرار داده است که می توانیم با کلیک روی آن ها از آن ها استفاده کنیم.

برای مثال set collection variable را می زنیم بعد در محل کدها خود پستمن یک تابع برای ما قرار می دهد که به شکل زیر است.

pm.collectionVariables.set("variable_key", "variable_value");

حالا ما در پارامتر اول یک نام به متغییر مورد نظر می دهیم و در پارامتر دوم مقداری که می خواهیم قبل از ارسال درخواست ست شود را قرار می دهیم.

pm.collectionVariables.set("postId", parseInt(Math.random() * 100));

حالا در بخش params به جای این که مقدار متغییر Path Variable  را دستی وارد کنیم، value را به شکل زیر قرار می دهیم.

 

URL   ==>  {{baseURL}}/posts/:post_id

Key   ==> post_id

Value   ==> {{postId}}

حالا زمانیکه که درخواست را ارسال می کنیم، هربار یک عدد رندوم برگشت داده می شود و پست جدیدی از سرور برمیگردد که در این صورت می توانیم بهتر تست های خود را انجام دهیم.


------ Test Api Writting ------


این بخش مربوط به بعد از ارسال درخواست است. یعنی بعد از اینکه response برگشت دستورات test اجرا می شوند. 

در این بخش هم باید کدهای جاوااسکریپت بنویسیم.

این بخش به شدت برای زمانی که باید توکن را دریافت و برای درخواست ها ست کنیم مناسب است. این مورد به ما کمک می کند که با هربار ارسال درخواست و تغییر توکن لاگین، توکن را دریافت کنیم و آن را ر یک متغییر بریزیم تا به راحتی در درخواست های دیگر از آن استفاده کنیم.

in test part

 

let res = pm.response.json();    // دسترسی به پاسخ 

if(res.id == 5){

    pm.collectionVariables.set("ali", "Vue");

}else{

    pm.collectionVariables.set("ali", "Laravel");

}

 

console.log(pm.collectionVariables.get("ali"));   // if == 5   ==> Vue    else  ===> Laravel

 

پستمن در بخش test یکسری توابع مانند status code 200 دارد که بررسی میکند آیا تمام مراحل ارسال و برگشت پاسخ به درستی بوده است یا نه. در این تابع ما یک مقدار تعریف می کنیم و یک callback function که اگر status code 200 بود فانکشن اجرا می شود و همچنین مقداری که به عنوان پارامتر اول داده ایم را به خروجی می فرستد.

برای دیدن خروجی حتما باید به تب test results  برویم.

pm.test("Status code is 200", function () {

    pm.response.to.have.status(200);

});

--   Monitor  Runner  -- 

 

می توانیم از منوی کناری monitoring را انتخاب کنیم و یک مانیتور بسازیم. وظیفه این مانیتور این است که می توانیم به آن بگوییم در بازه های زمانی مشخص به کالکشن ما درخواست ارسال کنید و گزارشی از آن در اختیار ما قرار دهیم.

تنها مشکلی که وجود دارد این است که monitoring روی localhost و لینک های مربوط به لوکال هاست کار نمی کند. یا به عبارت دیگر لوکال هایت لینک ها را نمی شناسد و ارور می دهد. این اقدام به این علت پیاده سازی شده است که همیشه localhost ران نیست و ممکن است در بازه زمانی که مانیتور درخواست ارسال می کند و ران نباشد و گزارشات با خطا مواجه شوند.

------ Mock Server ------


Mock server یک fake Api مانند json placeholder است.

زمانی که توسعه دهنده فرانت در حال کدنویسی است ممکن است هنوز server و backend آماده نباشد تا بتواند کدهایی که نوشته است را تست کند. برای این منظور ما می توانیم به راحتی از mock server استفاده کنیم.

به این صورت که ما یک سری end point با فیلدهای دیتایی که نیاز داریم ایجاد می کنیم و بعد آن را دیزاین می کنیم و می گوییم که اگر درخواست آمد مثلا فلان دیتا فیک را برگردان. در صورت به راحتی می توانیم کدهای فرانت را تست کنیم تا قسمت سرور و backend ایجاد شود.

این مورد یکی از بهترین ویژگی های postman است.


برای ساخت mock server کافی است از منوی کناری وارد بخش Mock شویم و روی create  بزنیم.

حالا مواردی که خواسته مانند url و متن جواب برگشتی و status code برگشتی و .. را انتخاب می کنیم و بعد باید کالکشن مورد نظر را انتخاب کنیم و روی next بزنیم.

------ Documention ------


زمانی که ما یک API می نویسیم باید حتما داکیومنت هم بنویسیم تا دیگران با خواندن آن داکیومنت کامل متوجه شوند که باید درخواست ها را با چه متودی و به چه شکل به چه URL ارسال کنند تا response مناسب را دریافت کنند. 

در حالت عادی نوشتن داکیومنت مدت زمانی خیلی طولانی از ما صرف می کند اما به کمک پسمتن می توانیم به راحتی در کنار تست API ها، برای آن ها یک داکیومنت عادی هم تولید کنیم.

اگر روی 3 نقطه کالکشن بزنیم به راحتی می توانیم گزینه view Documnetion را بزنیم تا پیش نمایشی از آن به ما داده شود. 

حالا اگر نیازی بود تغییرات را روی آن اعمال می کنیم.

حتی می توانیم در قسمت language  زبان برنامه نویسی که باید بعدا در خروجی پیش نمایش کدها برای آن باشد را انتخاب کنیم. برای مثال jQuery  or Node Axios

 

وقتی تنظیمات را لحاظ کردیم روی publish  می زنیم. حالا وارد یک صفحه می شود که می توانیم یکسری تنظیمات دیگر لحاظ کنیم مثل URL  اختصاصی - انتخاب Environment 

یکی دیگر از ویژگی ها این است که می توانیم رنگ بخش های مختلف doc خروجی را هم تنظیم کنیم.

تنظیمات را که لحاظ کردیم روی publish collection می زنیم.

حالا یک URL به ما می دهد که doc به طور کامل در آن قرار داده و به راحتی می توانیم آن را در اختیار هر کسی که خواستیم قرار بدهیم.

این URL فعال است و می توان از اطلاعات آن استفاده کرد تا این که آن را  unpublish کنیم. در این صورت آن URL غیرفعال می شود و با رفتن به آن 404 نمایش داده می شود.

همچنین هر وقت که خواستیم می توانیم از بالای صفحه بخش published تغییرات دیگربه doc اضافه کنیم یا لینک آن را کپی کنیم.

 


--  swagger   and    APIs  --


داکیومنت هایی که برای یک API نوشته می شود باید طبق یک فرمت مشخص باشد که هم انسان و هم ماشین بتواند آن را بخواند.

swagger یکسری ابزار است که به ما کمک می کند تا استانداردترین داکیومنت را بنویسیم.

این ابزار API ها را براساس OpenApi با استانداردترین فرمت در اختیار ما قرار می دهد که فرمت آن json یا YAML است و به راحتی می توانیم آن را به پستمن بدهیم تا خودش هم end point ها و درخواست های مورد نیاز ما را ایجاد کند.

برای استفاده از این بخش در postman می توانیم در منوی کناری به بخش APIs برویم. بعد create را بزنیم. حالا می گوید که یک نام انتخاب کن. سپس ورژن را می گوید وارد کند و بعد باید نوع را انتخاب کنیم که ما روی Open Api  می گذاریم و فرمت را می توانیم از نوع json  یا YAML انتخاب کنیم. حالا روی create می زنیم تا ساخته شود.

 

حالا ساخته شده است و می توانیم روی view Schema بزنیم و نمونه آن را ببنیم.  تغییرات لازم را روی آن لحاظ کنیم.
