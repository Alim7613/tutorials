Git

در فرایند انجام یک پروژه بارها و بارها در پروژه تغییرات اعمال می کنیم تا به نتیجه مطلوب برسیم. گاهی هم نیاز است برای بهبود روند انجام یک پروژه نیاز است که بخش های مختلف یک پروژه را در اختیار توسعه دهندگان مختلف قرار دهیم. گیت برای حل این مشکلات به ما کمک می کند.

git یک version controller است. یک داکیومنت از گیت در آدرس git-scm.com وجود دارد که می توانیم از آن استفاده کنیم. 

 

یک از ویژگی های عالی گیت این است که برای مثال ما یک تغییر در پروژه خود داده ایم اما نمی خواهیم آن تغییر باشد و می خواهیم برگردیم به تغییرات قبلی و کدهای قبلی بدون آن که تغییرات فعلی روی آن ها لحاظ شده باشد. گیت به خوبی این امکان را در اختیار ما قرار می دهد.

 

در پروژه های بزرگ شاید در یک برحه از زمان یک خروجی از کدها بگیریم اما بعد از مدتی از ما خواسته می شوید که توسعه دیگری روی پروژه انجام دهیم. قبلا باید می آمدیم پوشه پروژه قبلی را کپی و در قالب پروژه جدید آن را توسعه می دادیم که این کار زیاد معقول نبود اما گیت به ما امکان می دهد در قالب همان پروژه اول اما در یک شاخه جدید تغییرات خود را اعمال کنیم.

اصلا سطح پروژه مهم نیست در هر پروژه ای برای کنترل بهتر تغییرات و توسعه می توانیم از git استفاده کنیم.

برای نصب گیت به سایت آن به آدرس git-scm.com برویم. خودش در کنار صفحه بهترین نسحه را به ما پیشنهاد می دهد اما می توانیم به بخش downloads برویم و اگر خواستیم نسخه های دیگر را هم دانلود کنیم.

برای آن که متوجه شویم گیت به درستی نصب شده است یا نه وارد cmd می شویم و دستور زیر می زنیم.

git --version

 

البته به جز این با نصب گیت خودش یک ابزار با نام git bash در اختیار ما قرار می دهد که می توانیم از آن استفاده کنیم. برای استفاده از این ابزار روی desktop راست کلیک می کنیم و آن را انتخاب می کنیم. git bash here

این ابزار در هر directory در دسترس است و تنها کافی است راست کلیک کرده و آن را انتخاب کنیم. این ابزار امکان دسترسی سریع به گیت را به ما می دهد و عالی است.

// in git bash

$ git --help
 

 

دستور بالا اطلاعات کاملی در مورد دستورات مختلف git در اختیار ما قرار می دهد.

 

برای استفاده از git در دایرکتوری پروژه به شکل زیر عمل می کنیم.

1. ابتدا در cmd به دایرکتوری مدنظر می رویم یا می توانیم با دستور mkdir یک پوشه بسازیم در هر جایی

2. زمانی که در پروژه مورد نظر قرار گرفتیم می توانیم با دستور زیر vs code را در آن دایرکتوری باز کنیم

code .

زمانی که گیت نصب باشد با هر ترمینال یا ابزار خط فرمانی به آن دسترسی داریم.

3. حالا باید گیت را به پروژه اضافه کنیم و در واقع یک repository گیت را داخل پروژه ایجاد کنیم تا فایل ها و دایرکتوری های پروژه را کنترل کند. برای این کار باید ابتدا دستور زیر را داخل پوشه پروژه بزنیم.

git  init

با زدن این دستور یک repository از گیت در پروژه ایجاد می شود که ما هر تغییری در پروژه ایجاد کنیم، گیت آن را دنبال می کند.

این دستور داخل پوشه پروژه یک پوشه مخفی با نام دات گیت .git ایجاد میکند. اگر این پوشه را نمی توانستیم ببنیم باید کاری کنیم که ویندوز پوشه های hidden را هم نشان دهد.

حالا گیت ایجاد شده است و کاری به آن پوشه هم نداریم و باید مخفی باشد اما هر تغییری که در پروژه ایجاد کنیم، گیت آن را تشخیص می دهد و به ما اطلاع می دهد که می توانیم آن را نهایی کنیم یا بزاریم به همان حالت باشد تا بعدا آن را اصلاح کنیم. برای دیدن این تغییرات بهتر است از vscode استفاده کنیم.

 

 

زمانی که گیت را به پوشه پروژه اضافه می کنیم در vscode گیت با تغییر رنگ فایل هایی که اضافه کردیم یا آن ها را تغییر داده ایم به ما می گوید که دارد پروژه را دنبال می کند.

به کمک دستور زیر می توانیم متوجه شویم که وضعیت پروژه ما به چه شکل است. ایا شاخه کاری ما خالی است یا پر یا فرایندی برای اجرا داریم یا نه

git status

زمانیکه یک فایل جدید ایجاد می کنیم، رنگ آن در vscode سبز می شود و یک U در مقابل آن قرار می دهد. این بدین معنی است که فایل ما هنوز Untracked است و هنوز نهایی نشده، پس گیت تغییراتی که در این فایل ایجاد کنیم را دنبال نمی کند.

زمانی که دستور status را می زنیم گیت در بخش Untracked به ما می گوید که چه فایل هایی تازه ایجاد شده اند و در سطح ابتدایی هستند و باید سطح آن ها را تغییر دهیم.

ما چند سطح در گیت داریم.

1. شاخه ابتدایی: گیت تنها می گوید که این فایل اضافه شده است و تغییرات آن را دنبال نمی کند.

2. شاخه کاری: به ما می گوید که چه فایل هایی را اضافه کردیم و چه فایل هایی را در آن ها تغییر ایجاد کردیم.

 

به کمک دستور زیر می توانیم فایل ها را تایید کنیم تا گیت بتواند آن ها را چک کند و آن ها را زیر نظر بگیرد و تغییرات آن ها را دنبال کند.

git  add  fileName

با زدن این دستور فایل وارد stage می شود و گیت می تواند آن را زیر نظر بگیرد. همچنین علامت U به  (Added) A در vscode تبدیل می شود.

 

گاهی تعداد فایل هایی که می خواهیم به شاخه stage اضافه کنیم زیاد است و نیازی نیست ما اسم تک تک آن ها را وارد کنیم. با کمک دستور زیر می توانیم تمام فایل های دایرکتوری مدنظر را به شاخه  unstage اضافه کنیم.

git  add  .

 

 

------ Commit ------

 

یکی از مزایای گیت این است که در هر مرحله می توانیم برگردیم به حالت های قبلی که توش قرار داشتیم. 

ما با دستور Add فایل ها را به مخزن گیت اضافه کردیم، به این روش commit کردن هم می گویند. زمانی که به مخزن گیت اضافه می کنیم فایل ها ر stage کردیم.

 

برای حذف کردن فایل ها از مخرن گیت یا به عبارت دیگر unstage کردن فایل ها وقتی دستور git status را می زنیم، خود گیت هم اشاره کرده است که باید از دستور زیر استفاده کنیم.

git  rm  --cached  filename

 

گاهی نیاز است تمام فایل هایی که داخل Stage قرار دارند را unstage کنیم که برای این کار از دستور زیر استفاده می کنیم.

git  rm  --cached  -r  .

 

به کمک دستور زیر می توانیم به گیت بگوییم که تغییراتی که تا کنون داده ام را با این نامی که بهت دادم در مخزن خودت ذخیره کن.

git  commit  -m "add project to git"

 

با زدن این دستور فایل هایی که داخل stage قرار دارند را commit می کنیم و به مخزن گیت اضافه می کنیم و در vscode رنگ فایل ها و علامت هایی که جلوی هر فایل بود از بین می رود.

حالا بعد از این دستور اگر status را بزنیم میگه که شاخه کاری شما خالی است و فایلی داخل stage ندارید.

حالا اگر هر تغییری در فایل هایی که Commit شده اند ایجاد کنیم git متوجه می شود و به ما با زرد کردن رنگ اسم فایل و قرار دادن (Modified) M رو به روی آن در vscode این موضوع را می گوید. Modified یعنی تغییر کرده یا اصلاح شده

حالا  اگر دستور status را بزنیم می بینیم که اسم فایل در بخش chnage not staged for commit اضافه شده است. حالا برای این که فایل دوباره کامیت شود باید ابتدا آن را Add کنیم به stage و دوباره دستور commit را بزنیم.

یک نمای کلی از سطح های گیت تا کنون 

unstage   --add-->  stage   --commit-->   repository git

 

 

------ restore ------

 

گاهی نیاز داریم که فایل هایی که حتی آن ها را commit کردیم و به repository گیت اضافه کردیم را دوباره unstage کنیم که در این صورت از دستور زیر استفاده می کنیم.

git  restore  --staged  fileName

git  restore  --staged  ali.php

 

مهم: همچنین یک دستور داریم که به کمک آن می توانیم برگردیم به تغییرات قبلی که فایل را با ان ها commit کرده بودیم. عملا می خواهیم تغییرات جدید را نادیده بگیریم

git  restore  fileName

 

 

------ diff ------

 

اگر یک فایل را حذف کردیم می توانیم با دستور زیر آن را بازگردانی کنیم.

git restore FileName

 

به کمک دستور زیر می توانیم یک فایل را به طور کامل حذف کنیم.

git rm  -f   fileName

 

به کمک دستور زیر می توانیم تغییرات که در هر فایل  بین هر commit وجود دارد را به ما نمایش می دهد. همچنین تغییرات هر شاخه کاری را هم نمایش می دهد.

git diff

 

همان طور که قبلا دیدیم زمانی که یک فایل را تغییر می دهیم، از حالت کامیت به modifed تغییر می کند اما می توانیم از دستور زیر برای طی چند مرحله به صورت یکجا استفاده کنیم. این دستور خود به stage اضافه می کند و بعد commit می کند.

-a  =>  منطور add هست

git commit -a  -m "save changes"

 

برای تمام دستورات git می توانیم از help-- استفاده کنیم. این دستور به صفحه مرورگر می رود و یک داکیومنت کامل از آن دستور به ما ارائه می دهد.

git  diff --help

git  add  --help

git  restore  --help

------ log ------

 

به کمک دستور زیر می توانیم یک log کامل از commit هایی که انجام دادیم و کارهای دیگر روی شاخه های پروژه را مشاهده کنیم.

git  log

 

با توجه به اینکه دستور بالا اطلاعات کاملی در مورد کسی که کامیت را انجام داده و زمان و ... در اختیار ما قرار می دهد، گاهی نیاز داریم که تنها اطلاعات مختصری از هر کامیت را ببینم و به عبارت دیگر مختصری از هر چیزی را در یک خط مشاهده کنیم که برای این کار از دستور زیر استفاده می کنیم.

git  log  --oneline

 

گاهی می خواهیم محدودیت ایجاد کنیم برای مثال تنها 2 کامیت آخری را نشان دهد که برای این کار از دستور زیر استفاده می کنیم.

git  log -1 --oneline

 

یکی از قدرمتندترین کاربردهای log این است که کامیت ها را همراه تغییرات موجود در هر فایل نمایش می دهد. برای این کار از دستور زیر استفاده می کنیم.

git  log  -p

------ reset ------

 

گاهی بعد از انجام کلی کار نیاز پیدا می کنیم که به حالت اولیه پروژه برگردیم. که برای اینکار از دستور زیر استفاده می کنیم. ابتدا باید دستور log را بزنیم و ایدی کامیت مورد نظر را کپی کنیم. به جای ایدی کامل می توانیم قسمت اول ایدی را بنویسیم.

git reset   idCommit  (e.g.  dbd9b40361acc97b8f77e583ecdd6f7867fbd95b  or  dbd9b )

 

ایدی هر commit یونیک است و اگر آن را داشته باشیم به راحتی می توانیم بین commit های مختلف جابه جا شویم. برای مثال در دستور بالا ما ایدی اولین کامیت را به دستور reset دادیم تا به اولین کامیت برگردد اما اگر ایدی آخرین کامیت را قبل از دستور بالا در log کپی کنیم به راحتی با زدن دستور بالا و زدن ایدی اخرین کامیت می توانیم به اخرین کامیت مجددا برگردیم. بعد از هر بار اجرای این دستور باید دستور restore را هم بزنیم.

 

اگر نخواستیم بعد از اجرا دستور بالا حتما دستور restore را هم بزنیم می توانیم از option زیر استفاده کنیم.

git   reset   --hard  idCommit

 

 

------ branch ------

 

در حالت عادی زمانی که نسخه های مختلفی از برنامه را توسعه می دهیم باید برای هر نسخه یک پوشه داشته باشیم و تغییرات را در آن اعمال کنیم اما زمانی که از گیت استفاده می کنیم. نیازی به این کار نیست و تنها کافی است برای هر نسخه یک شاخه مجزا ایجاد کنیم.

یک دیگر از ویژگی های branch این است که گاهی نیاز داریم در طول توسعه یک پروژه توسعه دهندگان مختلفی روی شاخه های مختلف پروژه کار کنند و تغییرات آن ها روی پروژه در branch ها مختلف ذخیره شود.

در نهایت تغییرات در branch ها مختلف را merge می کنیم و نسخه نهایی را منتشر می کنیم. 

برای ایجاد branch جدید می توانیم از دستور زیر استفاده کنیم.

git  branch dev1

 

برای مشاهده کل branch هایی که داریم می توانیم از دستور زیر استفاده کنیم.

git  branch

//OR

git  branch -a

برای جابه جا شدن بین branch ها مختلف می توانیم از دستور زیر استفاده کنیم.

git  checkout  branchName

git  checkout  dev1

 

برای حذف یک branch  از دستور زیر استفاده می کنیم. اما توجه داشته باشید که branch فعال یا همان branch که در آن قرار داریم را نمی توانیم حذف کنیم.

git  branch -d  branchName

git  branch -d  div1

 

 

گاهی می خواهیم یک شاخه ایجاد کنیم و بلافاصله به آن برویم و نیازی نباشد که دو دستور بزنیم.

git  checkout  -b   branchName

 

 

------ merge ------

 

زمانی که از branch در گیت استفاده می کنیم تغییراتی که در هر فایل اضافه می کنیم تنها و تنها مربوط به همان branch هست.
برای مثال ما در شاخه Ali هستیم و در فایل index.php یک عبارت Aaaa را اضافه می کنیم. حالا اگر به branch دیگری فرضا MM سوئیچ کنیم، دیگر عبارت Aaaa در فایل index وجود ندارد. اما اگر دوباره به Ali برگردیم بازم این عبارت Aaaa را در فابل index می بینیم. 

این یک برتری برای گیت است که این امکان را فراهم می کند که چند توسعه دهنده به طور همزمان روی یک پروژه کار کنند. و در نهایت تغیرات همه فایل ها را merge می کنیم تا نسخه نهایی release شود. 

برای merge کردن باید به branch برویم که می خواهیم یکسری branch های دیگر را با آن ادغام کنیم.

//  in Desired branch 

git  merge  branchName     => شاخه که می خواهیم با شاخه ای که در آن هستیم ادغام شود.

git  merge  dev2

 

 

------ Stash ------

 

در گیت می توانیم ببینیم که کامیت ها چگونه بودند و همچنین مواردی که آن ها را merge کردیم چه چیزهایی بوده اند. برای این کار از دستور زیر استفاده می کنیم.

git  log  --graph

 

زمانی که تغییری روی فایل های یک branch ایجاد می کنیم، آن ها در حالت modified قرار می گیرند. حالا اگر بخواهیم از این شاخه به شاخه ای دیگر سوئیچ کنیم، گیت به ما می گوید که که یا باید تغییرات را restore کنیم و به حالت قبل برگردانیم یا آن ها را کامیت کنیم و از حالت modified در بیاوریم و یا این که از stash استفاده کنیم.

در حقیقت stash یک حالت کش کردن دارد که تغییرات ما را کش می کند و هر موقع که خواستیم استش را فراخوانی و از تغییراتی که کش کردیم استفاده می کنیم. برای این منظور از دستور زیر استفاده می کنیم.

git  stash

 

حالا اگر دستور زیر را بزنیم لیست stash ها را به ما می دهد.

git  stash  list

 

حالا برای اینکه ببنیم هر stash مربوط به چی هست و شامل چه اطلاعاتی است از دستور زیر استفاده می کنیم اما قبل از آن باید دستور stash list را زده باشیم و ایدی stash مورد نظر را کپی کنیم.

git  stash show   idStash

git  stash  show   stash@{0}

 

برای مشاهده این استش ها با جزئیات بیشتر و اطلاعات کامل در مورد تغییرات و ... از آپشتن p- استفاده می کنیم.

git  stash  show   -p   idStash

git  stash  show   -p   stash@{0}

 

برای ایجاد یک Stash سفارشی از دستور زیر استفاده می کنیم.

git  stash -m "message"

 

برای جذف یک stash از دستور زیر استفاده می کنیم.

git  stash  drop   idStash

 

برای فراخوانی تغییرات هر استش و استفاده از آن ها به شکل زیر عمل می کنیم. بعد از فراخوانی حتما باید کامیت را هم انجام دهیم.

git  stash apply  idStash

// OR

git  stash pop idStash

 

git commit  -a -m "message"

 

تفاوت apply  و pop

در حقیقت دستور Apply این امکان را به ما می دهد که استش مورد نظر را فراخوانی کنیم بدون آن که استش مورد نظر حذف شود. یعنی بعد از فراخوانی بازهم استش در لیست استش ها وجود دارد اما دستور pop بعد از فراخوانی استش، آنرا از لیست stash ها حذف می کند.

 

 

------ gitignore ------

 

گاهی نیاز است که به گیت بفهمانیم که یکسری فایل ها را اصلا کاری بهشون نداشته باش و آنها را دنبال نکن. برای این کار از فایل gitignore. استفاده می کنیم. 

برای اضافه کردن این فایل به پروژه باید حتما از git bash استفاده کنیم و به دایکتوری پروژه برویم و با راست کلیک آن را انتخاب کنیم.

// in git bash

touch .gitignore

git add .gitignore

git commit  -m "add .gitignore"

 

حالا داخل این فایل باید لیستی از فایل ها که نمی خواهیم گیت آن ها را دنبال کند، قرار دهیم.

زمانی که یک فایل جدید ایجاد می کنیم نمی توانیم مستقیما برای اولین بار از دستور

 git commit -a  -m "save changes"

استفاده کنیم و برای اولین بار باید اضافه شده به stage و commit را دستی انجام دهیم. 

 

زمانی که پوشه ای را به gitignore اضافه می کنیم اگر در ابتدای آن \ قرار دهیم تمام دایرکتوری ها و فایل های داخل آن را هم نادیده می گیرد.

گاهی ما فایل های زیادی از نوع txt داریم و می خواهیم همه آن ها را فیلتر کنیم و کاری کنیم که گیت توجهی به آن ها نداشته باشد. یا باید همه آن ها را به این فایل اضافه کنیم یا از دستور زیر برای فیلتر کردن همه فایل های txt  استفاده کنیم.

// in  gitignore

*.txt

 

گاهی می خواهیم بگوییم که همه فایل های txt را فیلتر کن به جز یک مورد خاص

*.txt

!name.txt

 

کامنت کردن در gitignore با #

گاهی قبلا فایل هایی را به Repository گیت اضافه کردیم اما حالا می خواهیم آن ها را نادیده بگیریم که برای این کار ابتدا فایل ها را از repository خارج می کنیم و بعد دوباره add را انجام می دهیم.

git  rm --cached -r

 

*** add file name to gitignore

 

git add .

 

 git commit -a  -m "second commit for gitignore"

 

 

------ git config ------

 

تا حالا هرچه گفته شده مربوط به کار با گیت روی لوکال بود اما نیاز است که گاها پروژه خود را به صورت عمومی یا private روی گیت هاب قرار دهیم تا سایر توسعه دهندگانی که روی این پروژه کار می کنند هم به آن دسترسی داشته باشند.

برای این کار نیاز به برخی از config ها داریم. کانفیگ هایی که انجام می دهیم داخل پوشه مخفی git. در پروژه و فایل config ذخیره می شود. (لوکال)

 

git config  user.name "ali mohseny"  ==> local

 

//Global

git config --global user.name "ali mohseny"

 

برای اپلود پروژه روی گیت هاب باید حتما 2 کانفیگ زیر را انجام دهید.

git config --global user.name "ali mohseny"

git config --global user.email  "ali@gmail.com"

 

 

------ github ------

 

برای اپلود پروژه روی گیت هاب باید بعد از لاگین یک repository ایجاد کنید.

بعد باید با دستور زیر یک remote ایجاد کنیم و بعد لینک مربوط به git clone ریپازیتوری که در گیت هاب ایجاد کردیم را به آن بدهیم.

git  remote add RemoteName https://github.com/Alim7613/test.git

git  remote add ali  https://github.com/Alim7613/test.git

 

حالا برای دیدن لیست همه remote هایی که داریم می توانیم از دستور زیر استفاده کنیم.

git  remote

 

برای حذف یک remote هم می توانیم از دستور زیر استفاده کنیم.

git  remote  remove  remoteName

git  remote  remove  ali

 

بعد از این برای اپلود باید از دستور push استفاده کنیم.

git  push -u remoteName branchName

 

 

اگر اطلاعاتی که در گیت کانفیگ ذخیره کردیم با اطلاعات حساب کاربری ما در سایت گیت هاب یکی باشد، گیت هاب ما را به عنوان یک توسعه دهنده اصلی پروژه شناسایی می کند.

برای اپدیت پروژه ای که روی گیت هاب قرار دادیم می توانیم ابتدا به branch مورد نظر در cmd برویم و بعد دستور زیر را بزنیم تا همچیز آپدیت شود.

git  push

 

 

------ pull - clone ------

 

گاهی می خواهیم عکس قضیه مربوط به اپدیت را انجام دهیم. یعنی بیاییم و تغییراتی که در سایت گیت هاب روی پروژه خود اعمال کردیم را روی پروژه خودمان روی سیستم لوکال هم اعمال کنیم. برای این کار می توانیم به روش زیر عمل کنیم.

git  pull  remoteName

 

گاهی به هر دلیل ممکن است پروژه از روی سیستم ما حذف شود که در این صورت نیاز پیدا می کنیم که دوباره آن را دانلود کنیم. می توانیم به راحتی آن را از گیت هاب دریافت کنیم. با دانلود کردن یا clone کردن.

در قسمت دانلود خود گیت هاب یک لینک را هم قرار می دهد که باید برای Clone کردن آن را برداریم و داخل git bash here در هر پوشه ای که خواستیم استفاده و دانلود کنیم پروژه را.

in desktop

in git bash here

git  clone  AddressCopied nameForFloderProjectDownloaded

 

اگر عملیات merge را روی پروژه انجام نداده باشیم زمانی که دانلود می کنیم تنها اطلاعات branch فعال که در گیت هاب در آن قرار داریم دانلود می شود اما اگر عملیات merge را انجام داده باشیم کل پروژه با تمام فایل های آن دانلود می شود.

 

برای merge کردن اطلاعات باید قبل از آپلود پروژه روی گیت هاب merge ها لازم بین branch ها مختلف را انجام دهیم و یک release نهایی تولید کنیم و آن را اپلود کنیم تا هنگام دانلود فایل های پروژه به درستی دانلود شوند.

این مزیت به ما کمک می کند بیشمار branch داشته باشیم اما هر زمان که خواستیم تنها مواردی که نیاز داریم را باهم merge کنیم و یک Release نهایی را تحت عنوان یک پروژه اپلود کنیم.

 

مزیت دیگر این است که ما می توانیم با ایجاد یک repository private از گیت هاب یک عنوان یک backup center استفاده کنیم و پروژه های خود را با تغییرات نهایی داخل آن قرار دهیم. مثلا بعد از هر بار توسعه پروژه دستور اپلود را بزنیم تا تغییرات همیشگی روی پروژه اعمال شوند و اگر سیستم شما به مشکلی خورد پروژه هایتان روی گیت هاب سالم باشند.

 

 

------ Fork ------

 

گاهی می خواهیم روی یک پروژه توسعه دهندگان دیگری هم کار کنند یا به عبارتی ما بیاییم و تیمی کار کنیم که در این صورت موضوع fork مطرح می شود.
